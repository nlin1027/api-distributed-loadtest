"A distributed load testing platform. Here is a breakdown of the goals and vision of the project:
Project vision: Build a distributed load testing platform — not just a tool that generates traffic, but a system with a fleet of coordinated worker processes (deployed via containers), each capable of generating load against a target API, orchestrated by a controller, observable through metrics on both the fleet and the target, able to automatically scale the fleet to match what a given test demands, and eventually capable of generating traffic from genuinely distinct network origins to simulate real multi-user, multi-IP traffic. This is a project meant to teach system design — each stage should be a self-contained project sitting on top of the last, with a concrete reason to exist rather than being infra for its own sake.

Stage 1 — Single-process engine. A single process that generates concurrent closed-loop virtual user load against a target URL, correctly recording outcomes (success/error/latency per request) and computing summary stats: throughput, p50/p95/p99 latency, error rate — with correct handling of empty or all-failed runs. Configurable via CLI arguments (target URL, user count, duration) rather than hardcoded values.
Stage 2 — Controller/worker split. The core distributed mechanics. The single-process engine becomes a reusable function callable either from a CLI or from a network handler. Each worker is a small HTTP server exposing an endpoint to accept a job (target URL, user count, duration) and return results. A controller knows the worker addresses, splits the total requested virtual users across them, dispatches the job to all workers concurrently, and collects their results — pooling each worker's raw latency data into one combined dataset and recomputing percentiles/throughput/error rate centrally, rather than averaging each worker's own already-computed stats. Handles partial failure: a timeout on collecting each worker's response, and a clear report of how many workers actually responded, rather than hanging or crashing outright on one bad worker.
Stage 3 — Containers. The worker is packaged as a Docker image — one reproducible artifact that runs identically anywhere. Several worker containers run locally via docker-compose to simulate a multi-machine fleet before touching real infrastructure, with the controller reaching workers by container name over Docker's internal networking rather than hardcoded host ports — the actual container-networking lesson of this stage.
Stage 4 — Metrics. Dual-purpose observability, tied into a live dashboard. Fleet health: each worker exposes its own internal state (CPU usage, virtual users currently assigned/sustained) via a Prometheus-style /metrics endpoint. Target health: the latency/error/throughput signal the engine already produces, exposed continuously (not just as an end-of-test summary) so it's visible live during a run. Both get scraped by Prometheus and visualized together in a dashboard, so fleet health and target health are visible side by side while a test is in progress.
Stage 5 — Autoscaling. Scoped to the test fleet, not the target under test. A control loop compares a test's demand (requested total virtual users) against the fleet's current capacity (worker count × how loaded each already is, using the fleet-health metrics from Stage 4). When demand exceeds capacity, it automatically starts more worker containers to close the gap, so hitting an ambitious concurrency target doesn't require manually pre-guessing fleet size. Can be hand-rolled (a small scaler watching a metric and calling the Docker API) or done via real Kubernetes HPA for the production-grade version of the same idea.
Stage 6 — Concurrent tests and load-aware autoscaling. The controller stops being a one-shot script and becomes a long-running service that accepts test jobs over HTTP (target URL, user count, duration), so multiple tests can run at the same time against different targets — which means workers can now be partially busy when a new job arrives. This breaks Stage 5's naive capacity math (total_users / max_users_per_worker, which assumes every worker is empty) and forces the real version from the original vision: capacity = per-worker headroom, computed from each worker's live worker_active_users (already exposed via /metrics), so a new job is placed onto existing spare capacity first and only the true shortfall triggers new containers. Prerequisite folded in here: the scaler must finally write Prometheus's target file (workers.json) on every fleet change, both because dynamic workers are currently invisible to Prometheus and because the stretch goal of this stage — trend-aware scaling decisions (e.g. scale on avg_over_time(worker_cpu_percent[2m]) via PromQL rather than an instantaneous read) — depends on Prometheus actually having the fleet's history. Scale-down must also get concurrency-safe: a worker is only reapable when no active test has users assigned to it.
Stage 7 — Distributed origin. (Unchanged from the original Stage 6.) Local containers, no matter how many, all share one machine's network egress — every request in the test fleet looks like it comes from a single IP, which is unrealistic for testing anything with per-IP defenses (rate limiting, geo-blocking, abuse detection) and doesn't reflect how real traffic from many distinct users actually arrives. This stage deploys worker containers to genuinely distinct network locations — different cloud regions and/or providers — so test traffic originates from diverse real IPs, the way traffic from real distributed users would. Requires securing the worker's job-acceptance endpoint (no longer just sitting on a private, trusted local network) and a way for the controller to know about workers that live outside any single Docker host.
Stage 8 — Frontend and test-run history. A custom web frontend replaces Grafana as the way to watch and review tests, built on two data paths that don't fully exist yet. First, richer instrumentation: metrics gain a test_id label so Prometheus's time-series history can be scoped to a specific run (today all runs blur into one continuous stream), and the per-worker gauges get upgraded where the type is wrong for the job — Counters for cumulative request/error totals, a Histogram for latency so distributions and quantiles are computed properly server-side across the whole fleet, rather than each worker overwriting three percentile gauges once a second. Second, results persistence: the controller's final aggregated report (throughput, p50/p95/p99, error rate) currently vanishes into stdout — it gets stored per-run (a small database or file store keyed by test ID) and served from a results API. The frontend then combines both: live dashboards during a run by querying Prometheus's HTTP API (query_range + PromQL — the same thing Grafana does under the hood), and a browsable history of past runs from the results store, with per-run Prometheus graphs scoped by test_id.
